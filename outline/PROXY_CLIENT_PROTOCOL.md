# MUDlark Proxy-Client Protocol Specification

**Version:** 1.0  
**Date:** November 2, 2025  
**Purpose:** Define the communication protocol between the iOS MUDlark client and the Go-based proxy relay server.

---

## Overview

The MUDlark proxy server acts as a persistent middleman between the iOS client and MUD servers. This allows the client to disconnect/reconnect freely (backgrounding, network changes, screen lock) while the proxy maintains the TCP connection to the MUD.

### Key Benefits
- **Persistent connection**: Proxy stays connected to MUD 24/7
- **Background resilience**: Client can disconnect and reconnect without losing session
- **Message buffering**: Missed messages are replayed on reconnection
- **IP privacy**: Optional multi-egress routing through VPN tunnels
- **Security**: WSS (TLS) encryption between client and proxy

---

## Transport Layer

### Protocol: WebSocket Secure (WSS)

- **Client connects to:** `wss://your-proxy-domain.com/ws`
- **TLS 1.2+** required for all connections
- **Certificate validation**: Client must validate server certificate
- **Port:** 443 (standard HTTPS/WSS port)

### Why WebSockets?
- Full-duplex communication (bidirectional streaming)
- Built-in message framing
- Native support in iOS via `URLSessionWebSocketTask`
- Firewall-friendly (runs over HTTPS port)
- Low overhead compared to polling

### Alternative Endpoints (Multi-Egress)
- `wss://your-proxy-domain.com/ws0` - Egress pool 0 (VPN exit 1)
- `wss://your-proxy-domain.com/ws1` - Egress pool 1 (VPN exit 2)
- `wss://your-proxy-domain.com/ws2` - Egress pool 2 (VPN exit 3)
- etc.

Client chooses endpoint based on user's egress preference setting.

Make sure to disable Nagle's algorithm.

## Authentication & Session Management

### JWT-Based Authentication

The client must authenticate every WebSocket connection using a JWT token.

#### JWT Requirements

**Token generation:**
- Generated by your auth service (could be same proxy server or separate)
- Algorithm: **HS256** (HMAC-SHA256) or **RS256** (RSA signatures)
- Must include standard claims and custom claims

**Standard JWT Claims:**
```json
{
  "iss": "mudlark-proxy",           // Issuer
  "aud": "mudlark-client",          // Audience
  "sub": "user_123abc",             // Subject (userId)
  "exp": 1730592000,                // Expiration (Unix timestamp)
  "iat": 1730588400                 // Issued at (Unix timestamp)
}
```

**Custom Claims:**
```json
{
  "sid": "session_xyz789",          // Session ID (persistent across reconnects)
  "mud_host": "mud.example.com",    // Optional: pre-authorized MUD host
  "mud_port": 4000,                 // Optional: pre-authorized MUD port
  "egress_pool": "auto"             // Optional: "auto", "0", "1", "2"
}
```

#### JWT Validation (Server-Side)

The proxy server must:
1. Verify signature using shared secret (HS256) or public key (RS256)
2. Check `aud` matches `"mudlark-client"`
3. Check `exp` has not passed (token not expired)
4. Extract `sub` → becomes `userId` for this connection
5. Extract `sid` → becomes `sessionId` for session binding
6. Reject connection if validation fails

#### Session Ownership Rules

- A `sessionId` is bound to the first `userId` that creates it
- Only that `userId` can reconnect to that `sessionId`
- If different `userId` attempts to use same `sessionId`: **reject with error**
- Multiple WebSocket connections from same `userId`+`sessionId` are allowed (e.g., iPad and iPhone)

## Message Format

All messages are **JSON-encoded WebSocket text frames**.

### Message Structure

```json
{
  "type": "messageType",
  "...additional fields..."
}
```

Every message has a `type` field that determines its purpose and structure.

## Client → Server Messages

### 1. `hello` - Initial Handshake

**Sent:** Immediately after WebSocket connection opens  
**Purpose:** Authenticate, identify session, request backlog replay

```json
{
  "type": "hello",
  "jwt": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "sessionId": "session_xyz789",
  "resumeFrom": 1234,
  "mudHost": "aardwolf.org",
  "mudPort": 4000,
  "mudTLS": false,
  "clientVersion": "1.0.0",
  "connectiontime": 3600
}
```

**Fields:**
- `jwt` (string, required): Authentication token
- `sessionId` (string, required): Persistent session identifier
  - If new session, generate a UUID client-side
  - If reconnecting, use same sessionId from previous connection
- `resumeFrom` (integer, required): Last `lineId` client received
  - Use `0` for new sessions or when not requesting backlog
  - Use last known `lineId` to request missed messages
- `mudHost` (string, conditional): MUD hostname/IP
  - Required if session doesn't exist yet
  - Optional if resuming (proxy already knows)
- `mudPort` (integer, conditional): MUD port
  - Required if session doesn't exist yet
  - Optional if resuming
- `mudTLS` (boolean, optional): Whether the proxy should connect to the MUD over TLS/SSL
  - Defaults to `false` (plain TCP)
  - Set to `true` for MUDs that advertise TLS on their port (e.g. port 4443)
  - Certificate verification is skipped; self-signed certs are accepted
- `clientVersion` (string, optional): Client version for logging/debugging
- `connectiontime` (integer, rqeuired): how long for proxy to stay connected to mud without flushing buffer
  - 0 can be used for "indefinite"

**Server Response:**
- On success: `status` message with `connected: true` (or `connecting: true`)
- On auth failure: `error` message, then close connection
- On session mismatch: `error` message about ownership, then close connection

---

### 2. `toMud` - Send Command to MUD

**Sent:** When user sends a command/text to the MUD  
**Purpose:** Forward text to MUD server

```json
{
  "type": "toMud",
  "text": "kill goblin"
}
```

**Fields:**
- `text` (string, required): Command to send to MUD
  - Client should NOT append `\n` - server will add it
  - Client should preserve leading/trailing spaces if intentional
  - Empty strings are allowed (sends blank line)

**Server Behavior:**
- Append `\n` (or `\r\n` if MUD requires CRLF)
- Forward immediately to MUD TCP socket
- If MUD connection is down: queue or drop (server decision)
- Do NOT echo back to client (MUD will echo if local echo is off)

**Sensitive Data Handling:**
- If proxy detects password prompt from MUD, it enters "sensitive mode"
- Next `toMud` message(s) will NOT be stored in ring buffer
- Prevents password from being logged or replayed

---

### 3. `ack` - Acknowledge Receipt

**Sent:** Periodically by client to confirm message processing  
**Purpose:** Allow server to trim ring buffer, track client state

```json
{
  "type": "ack",
  "lineId": 1250
}
```

**Fields:**
- `lineId` (integer, required): Highest `lineId` client has successfully processed

**Server Behavior:**
- Can safely discard lines older than acknowledged `lineId` from buffer
- Tracks "client is alive and processing" for monitoring
- Optional implementation detail (not critical for MVP)

---

### 4. `ping` - Keep-Alive (Optional)

**Sent:** Every 30-60 seconds when idle  
**Purpose:** Detect dead connections, prevent proxy timeout
- Proxy servery CANNOT send any blank commands to a MUD, this could be seen as botting.

```json
{
  "type": "ping",
  "timestamp": 1730588400
}
```

**Fields:**
- `timestamp` (integer, optional): Client's current Unix timestamp

**Server Response:**
- Send `pong` message with same timestamp

**Note:** WebSocket protocol has built-in ping/pong frames, but application-level ping is useful for debugging and can carry data.

---

### 5. `disconnect` - Graceful Shutdown

**Sent:** When user explicitly disconnects from MUD (not just backgrounding)  
**Purpose:** Signal proxy to close MUD connection and clean up session

```json
{
  "type": "disconnect",
  "reason": "user_logout"
}
```

**Fields:**
- `reason` (string, optional): Why disconnecting
  - `"user_logout"` - User clicked disconnect
  - `"switching_worlds"` - User switching to different MUD
  - `"app_closing"` - App being force-quit

**Server Behavior:**
- Close TCP connection to MUD
- Clear ring buffer from memory
- Send final `status` message with `connected: false`
- Close WebSocket connection
- Destroy session (sessionId becomes invalid)

**Important:** If client just closes WebSocket without `disconnect`, proxy should:
- Keep MUD connection alive (assume backgrounding/network drop)
- Keep session alive for reconnection
- Apply idle timeout policy (e.g., disconnect after 30 minutes of no client)

## Server → Client Messages

### 1. `status` - Connection Status Update

**Sent:** After `hello`, on MUD connect/disconnect, periodically  
**Purpose:** Inform client of current connection state

```json
{
  "type": "status",
  "connected": true,
  "mudHost": "aardwolf.org",
  "mudPort": 4000,
  "sessionId": "session_xyz789",
  "egressPool": "1",
  "uptime": 3600,
  "timestamp": 1730588400
}
```

**Fields:**
- `connected` (boolean, required): True if proxy has active TCP connection to MUD
- `mudHost` (string, required): Current MUD hostname
- `mudPort` (integer, required): Current MUD port
- `sessionId` (string, required): Session ID for this connection
- `egressPool` (string, optional): Which egress pool is being used ("0", "1", "auto")
- `uptime` (integer, optional): Seconds since MUD connection established
- `timestamp` (integer, required): Server Unix timestamp

**When Sent:**
- After successful `hello` validation
- When MUD connection completes
- When MUD connection drops (with `connected: false`)
- Periodically (e.g., every 60 seconds) for health monitoring

---

### 2. `fromMud` - Text from MUD

**Sent:** When proxy receives data from MUD  
**Purpose:** Deliver MUD output to client

```json
{
  "type": "fromMud",
  "lineId": 1251,
  "text": "The goblin snarls in pain!",
  "timestamp": 1730588401
}
```

**Fields:**
- `lineId` (integer, required): Monotonically increasing line counter
  - Starts at 1 for new session
  - Increments with each line from MUD
  - Never decreases or resets during session lifetime
  - Unique per session
- `text` (string, required): Raw line from MUD
  - May contain ANSI escape codes (client handles parsing)
  - May be empty string (blank line)
  - Preserves all whitespace/formatting
- `timestamp` (integer, required): Unix timestamp when received

**Client Behavior:**
- Parse ANSI codes and render styled text
- Store `lineId` as "last seen" for `resumeFrom` on next connect
- Append to scrollback buffer
- Process triggers, aliases, GMCP, etc.

**Server Behavior:**
- Split MUD data stream on newlines (`\n` or `\r\n`)
- Assign sequential `lineId` to each line
- Store in ring buffer (unless in sensitive mode)
- Broadcast to all connected WebSocket clients for this session

---

### 3. `backlog` - Replayed Messages

**Sent:** After `hello` if client specified `resumeFrom > 0`  
**Purpose:** Deliver messages client missed while disconnected

```json
{
  "type": "backlog",
  "lines": [
    {
      "lineId": 1235,
      "text": "A goblin enters from the north.",
      "timestamp": 1730588350
    },
    {
      "lineId": 1236,
      "text": "The goblin attacks you!",
      "timestamp": 1730588351
    }
  ],
  "resumedFrom": 1234,
  "currentLineId": 1250
}
```

**Fields:**
- `lines` (array, required): Array of missed messages
  - Each element has same structure as `fromMud` (lineId, text, timestamp)
  - Ordered by `lineId` (oldest first)
  - May be empty array if no missed messages
- `resumedFrom` (integer, required): The `lineId` client requested
- `currentLineId` (integer, required): Most recent lineId (tip of buffer)

**Client Behavior:**
- Render backlog messages (perhaps with visual indicator like timestamp or "--- missed ---")
- Update `lastSeenLineId` to `currentLineId`
- Continue receiving `fromMud` messages with `lineId > currentLineId`

**Server Behavior:**
- Search ring buffer for all lines where `lineId > resumeFrom`
- Cap replay to N lines (e.g., 2000) to prevent overwhelming client
- Send as single `backlog` message
- Immediately switch to live streaming (`fromMud` messages)

**Note:** If `resumeFrom` is too old (line no longer in buffer), server sends what it has. Client sees gap in lineIds and knows some messages were lost.

---

### 4. `error` - Error Notification

**Sent:** When something goes wrong  
**Purpose:** Inform client of errors, authentication failures, etc.

```json
{
  "type": "error",
  "code": "AUTH_FAILED",
  "message": "JWT signature verification failed",
  "fatal": true,
  "timestamp": 1730588400
}
```

**Fields:**
- `code` (string, required): Machine-readable error code
- `message` (string, required): Human-readable error description
- `fatal` (boolean, required): If true, connection will be closed
- `timestamp` (integer, required): Unix timestamp

**Error Codes:**

| Code | Description | Fatal | Reason |
|------|-------------|-------|--------|
| `AUTH_FAILED` | JWT validation failed | Yes | Invalid signature, expired token |
| `SESSION_CONFLICT` | SessionId belongs to different user | Yes | Session ownership mismatch |
| `INVALID_MESSAGE` | Malformed JSON or missing fields | No | Bad client message |
| `MUD_UNREACHABLE` | Cannot connect to MUD host | No | Network issue, bad host/port |
| `MUD_DISCONNECTED` | MUD closed connection | No | Server reboot, kicked, timeout |
| `RATE_LIMIT` | Too many requests | No | Anti-spam protection |
| `SERVER_ERROR` | Internal server error | Maybe | Unexpected server issue |

**Client Behavior:**
- Display error to user (especially if fatal)
- If `fatal: true`, don't attempt to reconnect immediately
- Log error for debugging
- If `MUD_DISCONNECTED`, proxy will auto-reconnect (wait for `status`)

---

### 5. `pong` - Keep-Alive Response

**Sent:** In response to client `ping`  
**Purpose:** Confirm connection is alive

```json
{
  "type": "pong",
  "timestamp": 1730588400
}
```

**Fields:**
- `timestamp` (integer, optional): Echo of client's ping timestamp

**Client Behavior:**
- Use to measure round-trip latency
- Detect connection issues if pong doesn't arrive within timeout

---

### 6. `systemMessage` - Server Notifications (Optional)

**Sent:** For server maintenance, warnings, etc.  
**Purpose:** Display important messages to user

```json
{
  "type": "systemMessage",
  "severity": "warning",
  "message": "Server will restart in 5 minutes for maintenance",
  "timestamp": 1730588400
}
```

**Fields:**
- `severity` (string, required): `"info"`, `"warning"`, `"error"`
- `message` (string, required): Message to display
- `timestamp` (integer, required): Unix timestamp

**Client Behavior:**
- Display as in-game system message (different color/style)
- Maybe show notification if severity is `"error"` or `"warning"`

## Security Considerations

### Client-Side Security

1. **Token Storage**
   - Store JWT securely using iOS Keychain
   - Never log JWT to console or files
   - Invalidate token on logout

2. **Certificate Validation**
   - Validate server TLS certificate
   - Pin certificate or public key for extra security (optional)
   - Reject self-signed certs in production

3. **Input Validation**
   - Validate all incoming JSON messages
   - Check `type` field exists and is expected value
   - Handle missing/unexpected fields gracefully

4. **Sensitive Data**
   - Don't log password commands locally
   - Clear password fields from memory after sending
   - Warn user that proxy can see traffic (privacy notice)

### Server-Side Security

1. **JWT Validation**
   - Verify signature on every `hello`
   - Check expiration strictly
   - Rate-limit failed auth attempts (per IP)

2. **Session Isolation**
   - Ensure sessionId cannot be guessed
   - Validate userId matches sessionId owner
   - Reject cross-session access attempts

3. **Password Detection**
   - Use regex to detect password prompts from MUD
   - Enter "sensitive mode" for 10 seconds
   - Don't buffer user input during sensitive mode

4. **Rate Limiting**
   - Limit `hello` attempts per IP (prevent token spray)
   - Limit `toMud` rate per session (prevent spam)
   - Limit concurrent connections per userId

5. **Memory Safety**
   - Ring buffer is memory-only (never disk)
   - Clear buffers on session destroy
   - Don't log line text to disk

## Ring Buffer Specification

The proxy maintains an in-memory ring buffer per session to support message replay.

### Requirements

- **Capacity:** Configurable, default 2000 lines
- **Storage:** RAM only, never persisted to disk
- **Structure:** Circular buffer (FIFO when full)
- **Per-Session:** Each sessionId has its own buffer
- **Thread-Safe:** Must support concurrent reads/writes

### Data Stored Per Line

```go
type BufferedLine struct {
    LineID    int64     // Monotonic counter
    Text      string    // Raw line from MUD
    Timestamp int64     // Unix timestamp
}
```

### Behavior

- When buffer is full, oldest line is dropped (FIFO)
- Lines are never modified after insertion
- `resumeFrom` query returns all lines where `LineID > resumeFrom`
- Cap replay to 2000 lines max (prevent overwhelming client)
- Clear entire buffer when session is destroyed

### Sensitive Mode

When proxy detects password prompt (regex: `(?i)password[:>\s]`):
- Set `sensitiveUntil = now + 10 seconds`
- During this window:
  - **Do NOT** buffer outgoing `toMud` messages
  - **Do NOT** buffer incoming MUD responses (may contain password echo)
  - Continue normal buffering after timeout

This prevents passwords from being stored or replayed.

### Expiration

- Clear entire buffer when session ends

## MUD Connection Behavior

### Initial Connection

1. Proxy receives `hello` with `mudHost` and `mudPort`
2. Opens TCP socket to `mudHost:mudPort`
3. Sets socket options:
   - `TCP_NODELAY = true` (disable Nagle's algorithm for low latency)
   - `SO_KEEPALIVE = true` (detect dead connections)
   - Optional: `SO_BINDTODEVICE` to specific VPN interface (multi-egress)
4. Sends `status` message to client when connected

### Keepalive Strategy

**TCP-Level:**
- `SO_KEEPALIVE` with system defaults (or custom intervals)
- Detects dead connections at network layer

**Application-Level (Optional):**
- Send periodic heartbeat to MUD (e.g., `"\n"` every 120 seconds)
- Configurable per-MUD (some MUDs dislike idle spam)
- Only send if no other activity in last N seconds

### Reconnection Logic

If MUD connection drops:
1. Send `error` message to client: `MUD_DISCONNECTED`
2. Send `status` message with `connected: false`
3. Only attempt to reconnect when the user sends a signal.

### Data Handling

**MUD → Client:**
- Read from TCP socket continuously
- Split on `\n` or `\r\n` (detect MUD's line ending)
- Assign `lineId` to each line
- Store in ring buffer (unless sensitive mode)
- Broadcast `fromMud` to all connected clients

**Client → MUD:**
- Receive `toMud` messages from WebSocket
- Append `\n` (or `\r\n` if MUD uses CRLF)
- Write immediately to TCP socket
- Handle write errors (connection down):
  - Queue message for retry, or
  - Drop and inform client via `error` message

## Multi-Egress (VPN Routing)

### Purpose

Allow users to route MUD connections through different VPN exits to:
- Avoid IP-based multi-play detection
- Distribute load across multiple exits
- Geographic diversity

### Implementation Approaches

#### Option 1: Multiple Server Instances (Simpler)

- Run N copies of proxy, each in its own network namespace
- Each instance bound to different VPN interface
- Listen on different internal ports (8080, 8081, 8082)
- Reverse proxy routes `/ws0` → 8080, `/ws1` → 8081, etc.

**Client behavior:**
- User selects "Egress Pool: Auto / 0 / 1 / 2"
- Client connects to `wss://domain.com/ws0` for pool 0
- Different WebSocket endpoint = different egress

#### Option 2: Single Process with SO_BINDTODEVICE (Advanced)

- One proxy process
- When creating MUD TCP socket, use `SO_BINDTODEVICE` to bind to specific interface
- Requires elevated privileges (CAP_NET_RAW)
- More complex but single process

**Client behavior:**
- User selects egress pool in settings
- Client includes `egressPool` in `hello` message
- Proxy binds outbound socket to requested interface

### Egress Pool Selection

Client sends preference in `hello`:
```json
{
  "type": "hello",
  "jwt": "...",
  "sessionId": "...",
  "egressPool": "auto",  // or "0", "1", "2"
  ...
}
```

- `"auto"` = server picks (round-robin or least-used)
- `"0"`, `"1"`, `"2"` = specific pool
- Invalid pool → fallback to "auto"

Server confirms in `status`:
```json
{
  "type": "status",
  "connected": true,
  "egressPool": "1",
  ...
}
```

## Error Handling & Edge Cases

### Connection Timeout

**Scenario:** Proxy cannot reach MUD within timeout  
**Behavior:**
- Send `error` with code `MUD_UNREACHABLE`
- Send `status` with `connected: false`
- Continue retry logic
- Keep WebSocket open (client may want to change MUD/port)

### Invalid JWT

**Scenario:** JWT signature invalid, expired, or wrong audience  
**Behavior:**
- Send `error` with code `AUTH_FAILED`, `fatal: true`
- Close WebSocket with code 4001 (unauthorized)
- Log attempt (possible attack)

### Session Conflict

**Scenario:** User A tries to connect to User B's session  
**Behavior:**
- Send `error` with code `SESSION_CONFLICT`, `fatal: true`
- Close WebSocket with code 4003 (forbidden)
- Log attempt with both userIds

### Malformed Message

**Scenario:** Client sends invalid JSON or missing required fields  
**Behavior:**
- Send `error` with code `INVALID_MESSAGE`, `fatal: false`
- Log the bad message
- Continue accepting new messages

### Buffer Overflow

**Scenario:** Ring buffer fills up, client `resumeFrom` too old  
**Behavior:**
- Send `backlog` with available lines only
- Client detects gap in lineIds (e.g., resumed from 100, first backlog line is 500)
- Client shows "--- some messages lost ---" indicator

### Idle Timeout

**Scenario:** No client connected for X+ minutes  
**Behavior:**
- Default of 1 hour
- Close MUD connection
- Store ring buffer for possible reconnect by user for up to 24 hours
- Clear ring buffer after 24 hours
- Destroy session after 24 hours
- Next `hello` from client creates new session (loses history)

### Rate Limiting

**Scenario:** Client sends too many `toMud` commands too fast  
**Behavior:**
- Send `error` with code `RATE_LIMIT`
- Drop excess messages
- Consider temporary ban if severe

## Client Implementation Checklist

### Connection Setup
- [ ] Generate or load JWT from secure storage
- [ ] Create UUID for `sessionId` (or reuse existing)
- [ ] Open WebSocket to appropriate `/ws` or `/wsN` endpoint
- [ ] Validate server TLS certificate
- [ ] Send `hello` immediately after WS connects

### Message Handling
- [ ] Parse incoming JSON messages by `type`
- [ ] Handle `status`, `fromMud`, `backlog`, `error`, `pong`, `systemMessage`
- [ ] Render ANSI codes from `fromMud` text
- [ ] Track last received `lineId` for `resumeFrom`
- [ ] Send `ack` periodically (optional but recommended)

### User Input
- [ ] Send `toMud` when user presses enter
- [ ] Handle command history (client-side)
- [ ] Process aliases and triggers (client-side)
- [ ] Don't log passwords locally

### Reconnection Logic
- [ ] Detect WebSocket disconnect (network change, background)
- [ ] Store last `lineId` before disconnect
- [ ] Reconnect with same `sessionId` and `resumeFrom`
- [ ] Handle `backlog` and resume live stream

### Graceful Shutdown
- [ ] Send `disconnect` when user explicitly disconnects
- [ ] Close WebSocket cleanly
- [ ] Clear sensitive data from memory

### Error Handling

## Appendix: Example Implementations

### Client Connection (Swift)

```swift
class ProxyConnection {
    private var webSocket: URLSessionWebSocketTask?
    private let jwt: String
    private let sessionId: String
    private var lastLineId: Int64 = 0
    
    func connect(mudHost: String, mudPort: Int) {
        let url = URL(string: "wss://your-proxy.com/ws")!
        webSocket = URLSession.shared.webSocketTask(with: url)
        webSocket?.resume()
        
        // Send hello
        let hello = [
            "type": "hello",
            "jwt": jwt,
            "sessionId": sessionId,
            "resumeFrom": lastLineId,
            "mudHost": mudHost,
            "mudPort": mudPort,
            "clientVersion": "1.0.0"
        ]
        
        sendJSON(hello)
        receiveMessage()
    }
    
    func sendCommand(_ text: String) {
        let message = [
            "type": "toMud",
            "text": text
        ]
        sendJSON(message)
    }
    
    private func receiveMessage() {
        webSocket?.receive { [weak self] result in
            switch result {
            case .success(let message):
                if case .string(let text) = message {
                    self?.handleJSON(text)
                }
                self?.receiveMessage() // Continue receiving
            case .failure(let error):
                print("WebSocket error: \(error)")
            }
        }
    }
    
    private func handleJSON(_ text: String) {
        guard let data = text.data(using: .utf8),
              let json = try? JSONSerialization.jsonObject(with: data) as? [String: Any],
              let type = json["type"] as? String else {
            return
        }
        
        switch type {
        case "fromMud":
            if let lineId = json["lineId"] as? Int64,
               let text = json["text"] as? String {
                lastLineId = lineId
                handleMudOutput(text)
            }
        case "backlog":
            if let lines = json["lines"] as? [[String: Any]] {
                for line in lines {
                    if let lineId = line["lineId"] as? Int64,
                       let text = line["text"] as? String {
                        lastLineId = lineId
                        handleMudOutput(text)
                    }
                }
            }
        case "status":
            handleStatus(json)
        case "error":
            handleError(json)
        default:
            break
        }
    }
}
```

### Server Session (Go Pseudocode)

```go
type Session struct {
    SessionID   string
    UserID      string
    MudConn     net.Conn
    WsClients   []*websocket.Conn
    RingBuffer  *RingBuffer
    LineCounter int64
    mu          sync.RWMutex
}

func (s *Session) HandleFromMud(line string) {
    s.mu.Lock()
    s.LineCounter++
    lineID := s.LineCounter
    
    // Store in buffer
    s.RingBuffer.Add(lineID, line, time.Now().Unix())
    
    // Broadcast to all WS clients
    msg := map[string]interface{}{
        "type":      "fromMud",
        "lineId":    lineID,
        "text":      line,
        "timestamp": time.Now().Unix(),
    }
    s.mu.Unlock()
    
    s.broadcastToClients(msg)
}

func (s *Session) HandleToMud(text string) {
    s.mu.RLock()
    defer s.mu.RUnlock()
    
    if s.MudConn != nil {
        s.MudConn.Write([]byte(text + "\n"))
    }
}
```

## Summary

This protocol provides:
- ✅ **Secure communication** via WSS (TLS)
- ✅ **Authentication** via JWT
- ✅ **Session persistence** across reconnections
- ✅ **Message replay** via ring buffer
- ✅ **Privacy** (memory-only, no disk logging)
- ✅ **Multi-egress** support for IP diversity
- ✅ **Simple JSON messages** for easy implementation
- ✅ **Raw text** from MUD (client handles ANSI parsing)

The client and server can be implemented independently as long as both follow this specification.
